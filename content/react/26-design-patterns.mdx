## Why this exists

As React applications grow, the hardest problems are often no longer about syntax. They are about structure. How should logic be shared? How should related components coordinate? How much UI should a component own? How do you build flexible systems without turning every component into a giant one-off file?

Design patterns exist to answer those recurring structural questions. They are not special React rules. They are proven ways of organizing React code so it stays reusable, understandable, and scalable. Different patterns solve different kinds of problems, and strong React engineering is often about choosing the right one instead of forcing one pattern everywhere.

This topic matters because large apps are not built from isolated components. They are built from systems of components. Design patterns help those systems stay clear instead of collapsing into ad hoc code.

## How it works

Composition over inheritance is the core React mindset. Instead of extending a base UI class and overriding behavior, you build complex UI from smaller parts. A card can be made from a header, body, and footer. A layout can be made from a sidebar, main content, and toolbar. This is more flexible than inheritance because pieces can be rearranged and reused independently.

Compound components are groups of components that work together as one unit, usually with shared state behind the scenes. Tabs, accordions, and selects are common examples. The parent provides coordination, and child components express the structure declaratively.

Render props share logic by passing a function that returns UI. The component with the logic calls that function and gives it the current state. This pattern was more common before custom hooks but is still useful to understand.

Higher-order components, or HOCs, are functions that take a component and return an enhanced component. They were a major reuse pattern before hooks and still appear in some codebases, especially around auth, logging, or cross-cutting concerns.

Controlled and uncontrolled components describe who owns the state. A controlled component receives value and change callbacks from outside. An uncontrolled component manages its own internal state. Some advanced components support both patterns.

The headless component pattern separates logic from presentation. A headless hook or component provides state, behavior, and accessibility rules, while the consumer provides the final markup and styling. This is powerful when teams need the same behavior across very different UIs.

The provider pattern uses context to inject shared values or services through part of the tree. Theme providers and auth providers are common examples.

The container/presentational pattern separates data-fetching or stateful orchestration from pure rendering components. The container handles logic. The presentational component focuses on the visual output. Modern React does not always enforce this split rigidly, but the idea is still useful for keeping code readable.

## Code examples

### Example 1 — Composition over inheritance

Build a card from smaller focused parts.

```tsx
function Card({ children }: { children: React.ReactNode }) {
  return <section className="card">{children}</section>
}

function CardHeader({ children }: { children: React.ReactNode }) {
  return <header className="card-header">{children}</header>
}

function CardBody({ children }: { children: React.ReactNode }) {
  return <div className="card-body">{children}</div>
}

function CardFooter({ children }: { children: React.ReactNode }) {
  return <footer className="card-footer">{children}</footer>
}

export default function ProfileCard() {
  return (
    <Card>
      <CardHeader>Profile</CardHeader>
      <CardBody>User details go here.</CardBody>
      <CardFooter>Last updated today.</CardFooter>
    </Card>
  )
}
```

### Example 2 — Compound components: Tabs

Related components work together through shared context.

```tsx
import { createContext, useContext, useState, type ReactNode } from 'react'

type TabsContextValue = {
  activeTab: string
  setActiveTab: (id: string) => void
}

const TabsContext = createContext<TabsContextValue | null>(null)

function useTabsContext() {
  const context = useContext(TabsContext)

  if (!context) {
    throw new Error('Tabs components must be used inside Tabs')
  }

  return context
}

function Tabs({
  defaultTab,
  children,
}: {
  defaultTab: string
  children: ReactNode
}) {
  const [activeTab, setActiveTab] = useState(defaultTab)

  return (
    <TabsContext.Provider value={{ activeTab, setActiveTab }}>
      {children}
    </TabsContext.Provider>
  )
}

function TabList({ children }: { children: ReactNode }) {
  return <div>{children}</div>
}

function Tab({ id, children }: { id: string; children: ReactNode }) {
  const { activeTab, setActiveTab } = useTabsContext()

  return (
    <button onClick={() => setActiveTab(id)} aria-selected={activeTab === id}>
      {children}
    </button>
  )
}

function TabPanel({
  whenActive,
  children,
}: {
  whenActive: string
  children: ReactNode
}) {
  const { activeTab } = useTabsContext()

  if (activeTab !== whenActive) return null

  return <div>{children}</div>
}
```

### Example 3 — Compound components: Accordion

The same idea works for different UI.

```tsx
import { useState } from 'react'

function AccordionItem({
  title,
  children,
}: {
  title: string
  children: React.ReactNode
}) {
  const [open, setOpen] = useState(false)

  return (
    <section>
      <button onClick={() => setOpen((value) => !value)}>{title}</button>
      {open ? <div>{children}</div> : null}
    </section>
  )
}

export default function Accordion() {
  return (
    <div>
      <AccordionItem title="What is React?">A UI library.</AccordionItem>
      <AccordionItem title="What are hooks?">Reusable stateful logic.</AccordionItem>
    </div>
  )
}
```

### Example 4 — Render props

Share logic by passing a function that receives state.

```tsx
import { useState } from 'react'

function MouseTracker({
  children,
}: {
  children: (position: { x: number; y: number }) => React.ReactNode
}) {
  const [position, setPosition] = useState({ x: 0, y: 0 })

  return (
    <div
      onMouseMove={(event) =>
        setPosition({ x: event.clientX, y: event.clientY })
      }
    >
      {children(position)}
    </div>
  )
}

function Example() {
  return (
    <MouseTracker>
      {({ x, y }) => <p>Mouse position: {x}, {y}</p>}
    </MouseTracker>
  )
}
```

### Example 5 — HOC: `withAuth`

A higher-order component wraps another component.

```tsx
import { Navigate } from 'react-router-dom'

function withAuth<T extends object>(
  Component: React.ComponentType<T>,
) {
  return function AuthenticatedComponent(props: T & { isAuthed: boolean }) {
    if (!props.isAuthed) {
      return <Navigate to="/login" replace />
    }

    const { isAuthed: _ignored, ...rest } = props
    return <Component {...(rest as T)} />
  }
}
```

### Example 6 — HOC: `withLogging`

HOCs can add cross-cutting behavior.

```tsx
function withLogging<T extends object>(
  Component: React.ComponentType<T>,
) {
  return function LoggedComponent(props: T) {
    console.log(`Rendered ${Component.displayName ?? Component.name ?? 'Component'}`)
    return <Component {...props} />
  }
}
```

### Example 7 — Controlled/uncontrolled component pattern

A component can support both styles.

```tsx
import { useState } from 'react'

type CalendarProps = {
  value?: string
  onChange?: (value: string) => void
}

function Calendar({ value, onChange }: CalendarProps) {
  const [internalValue, setInternalValue] = useState('2026-04-12')

  const selectedDate = value ?? internalValue

  function handleChange(nextValue: string) {
    if (onChange) {
      onChange(nextValue)
    } else {
      setInternalValue(nextValue)
    }
  }

  return (
    <input
      type="date"
      value={selectedDate}
      onChange={(event) => handleChange(event.target.value)}
    />
  )
}
```

### Example 8 — Headless component pattern

Logic without UI gives consumers full control over presentation.

```tsx
import { useMemo, useState } from 'react'

function useCombobox(options: string[]) {
  const [query, setQuery] = useState('')
  const [open, setOpen] = useState(false)

  const filteredOptions = useMemo(() => {
    return options.filter((option) =>
      option.toLowerCase().includes(query.toLowerCase()),
    )
  }, [options, query])

  return {
    query,
    setQuery,
    open,
    setOpen,
    filteredOptions,
  }
}

function SearchableList() {
  const combobox = useCombobox(['React', 'Next.js', 'JavaScript'])

  return (
    <section>
      <input
        value={combobox.query}
        onChange={(event) => combobox.setQuery(event.target.value)}
        onFocus={() => combobox.setOpen(true)}
      />
      {combobox.open ? (
        <ul>
          {combobox.filteredOptions.map((option) => (
            <li key={option}>{option}</li>
          ))}
        </ul>
      ) : null}
    </section>
  )
}
```

### Example 9 — Provider pattern

Providers inject shared dependencies or shared state.

```tsx
import { createContext, useContext, useState, type ReactNode } from 'react'

type Theme = 'light' | 'dark'

type ThemeContextValue = {
  theme: Theme
  toggleTheme: () => void
}

const ThemeContext = createContext<ThemeContextValue | null>(null)

function ThemeProvider({ children }: { children: ReactNode }) {
  const [theme, setTheme] = useState<Theme>('light')

  function toggleTheme() {
    setTheme((current) => (current === 'light' ? 'dark' : 'light'))
  }

  return (
    <ThemeContext.Provider value={{ theme, toggleTheme }}>
      {children}
    </ThemeContext.Provider>
  )
}
```

### Example 10 — Container/presentational pattern

Separate orchestration from display.

```tsx
import { useEffect, useState } from 'react'

type User = {
  id: number
  name: string
}

function UserCard({ user }: { user: User }) {
  return <article>{user.name}</article>
}

function UserListContainer() {
  const [users, setUsers] = useState<User[]>([])

  useEffect(() => {
    async function loadUsers() {
      const response = await fetch('/api/users')
      const data = (await response.json()) as User[]
      setUsers(data)
    }

    void loadUsers()
  }, [])

  return (
    <section>
      {users.map((user) => (
        <UserCard key={user.id} user={user} />
      ))}
    </section>
  )
}
```

The container handles fetching. The presentational component handles the UI shape.

## Common mistakes

<Callout type="mistake">
**Mistake:** Treating patterns as rules that must be used everywhere.
**Fix:** Choose the pattern that solves the current structural problem best.
**Why:** Patterns are tools, not laws. Overusing a pattern can make simple code more complex.
</Callout>

<Callout type="mistake">
**Mistake:** Reaching for inheritance-style thinking in React UI.
**Fix:** Prefer composition and smaller reusable pieces.
**Why:** Composition is more flexible and better aligned with how React builds interfaces.
</Callout>

## How this is used in real projects

Design patterns show up when products move beyond isolated pages into reusable systems. Compound components help design-system primitives like Tabs and Accordion. Headless patterns power accessible but fully stylable building blocks. Providers manage shared concerns like theme and auth. Container/presentational separation helps keep data-fetching code from leaking into pure UI components.

No single app uses all of these patterns equally, and many modern codebases mix them together. A team might use composition for most UI, headless hooks for advanced reusable behavior, providers for shared concerns, and HOCs only in a few older areas. The skill is recognizing what problem each pattern is trying to solve.

## Interview questions

<Callout type="interview">
**Q:** What is composition over inheritance?
**A:** Composition over inheritance means building complex UI from smaller reusable pieces instead of extending a base class or hierarchy. In React, this is the standard way to structure reusable components because it is more flexible and easier to combine in different ways.
</Callout>

<Callout type="interview">
**Q:** What are compound components?
**A:** Compound components are related components that work together as a coordinated family, often through shared state or context. Tabs, Accordions, and Select components are common examples.
</Callout>

<Callout type="interview">
**Q:** What is a higher-order component?
**A:** A higher-order component is a function that takes a component and returns a new component with added behavior or injected props. It was a common reuse pattern before hooks and still appears in some codebases.
</Callout>

<Callout type="interview">
**Q:** What is the headless component pattern?
**A:** The headless component pattern separates logic and behavior from presentation. A headless hook or component manages state and rules, while the consuming component decides the actual markup and styling.
</Callout>

<Callout type="interview">
**Q:** What is the difference between render props and custom hooks for sharing logic?
**A:** Render props share logic by passing a function that receives state and returns UI. Custom hooks share logic by returning state and behavior directly to the component. Custom hooks are usually cleaner in modern React because they avoid extra wrapper components in the tree.
</Callout>
