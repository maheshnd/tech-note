## Why this exists

Before React, building interactive user interfaces often meant manually finding DOM elements, changing their text, adding classes, hiding sections, and remembering every place that needed to update when data changed. That approach works for tiny pages, but it gets hard to manage as soon as the UI has many moving parts like counters, forms, lists, tabs, and dashboards. React was created to solve that problem by giving you a better mental model for UI: instead of manually updating the page step by step, you describe what the UI should look like for the current data. React then figures out how to update the browser efficiently and predictably.

React also helps developers stop thinking in terms of one giant page file and start thinking in reusable pieces. A real app is rarely just one screen. It is usually made of headers, cards, tables, forms, sidebars, buttons, modals, and status messages. React makes those pieces easy to split into components so the code is easier to read, reuse, and maintain.

## How it works

React is a JavaScript library for building user interfaces. It is a library, not a full framework. That means React focuses mainly on the view layer: rendering UI, updating UI when data changes, and organizing the UI into components. It does not, by itself, give you routing, deployment rules, backend APIs, or global data-fetching conventions. Those usually come from other tools such as React Router, TanStack Query, or frameworks like Next.js.

The core idea in React is declarative UI. In an imperative approach, you tell the browser exactly what to do in order: find this element, change this text, add this class, remove that node. In React, you describe the result you want for the current state of your data. If the count is `3`, show `Count: 3`. If the user is logged in, show the dashboard. If there are no results, show the empty state. React takes that description and updates the real DOM to match it.

React does this with a concept often called the Virtual DOM. The Virtual DOM is not the real browser DOM. It is a lightweight JavaScript representation of what the UI should look like. When data changes, React creates a new virtual tree, compares it with the previous one, and calculates the minimum set of real DOM changes needed. The biggest beginner takeaway is not "Virtual DOM is magic." The real lesson is that React gives you predictable UI updates without forcing you to manually mutate the page.

React also encourages component-based architecture. Instead of writing one massive file for a full screen, you break the UI into smaller reusable parts such as `Header`, `Sidebar`, `Main`, `Footer`, `DashboardCard`, or `PolicyStatusBadge`. This matters because real applications grow quickly. Reusable UI pieces reduce duplication, make bugs easier to isolate, and make teams faster.

React is also commonly used to build Single Page Applications, often called SPAs. In a traditional multi-page website, clicking a link usually sends a request to the server, which returns a brand new HTML page. In a SPA, the browser loads one main app shell and then JavaScript updates the visible screen without full page reloads for most interactions. That can make the experience feel smoother and more app-like. React itself does not define the routing layer, but it is commonly used to build SPAs.

Finally, React only handles the view layer in the bigger frontend picture. It is concerned with rendering UI and responding to state changes. Things like routing, server rendering, authentication flows, image optimization, and deployment conventions come from other tools layered around React.

## Code examples

### Example 1 — Traditional DOM manipulation vs React

One of the clearest ways to understand React is to compare it with direct DOM manipulation. Both examples below update a counter, but the mental model is very different.

```js
// Traditional DOM manipulation
const button = document.querySelector('#increment')
const output = document.querySelector('#count')

let count = 0

button.addEventListener('click', () => {
  count += 1
  output.textContent = `Count: ${count}`
})
```

```tsx
import { useState } from 'react'

export default function Counter() {
  const [count, setCount] = useState(0)

  return (
    <button onClick={() => setCount((value) => value + 1)}>
      Count: {count}
    </button>
  )
}
```

In the DOM version, you are responsible for updating the page yourself. In the React version, you update state and React updates the UI for you. That is the shift from imperative UI to declarative UI.

### Example 2 — Your first React component

A React component is just a JavaScript function that returns JSX.

```tsx
export default function HelloWorld() {
  return <h1>Hello from React</h1>
}
```

This is the smallest useful React component. It already shows the basic model: a component is a reusable UI function.

### Example 3 — Rendering a component into the DOM with `createRoot`

React needs one real DOM element to attach to. That element usually lives in `index.html`.

```tsx
import React from 'react'
import ReactDOM from 'react-dom/client'
import App from './App'

const container = document.getElementById('root')

if (!container) {
  throw new Error('Root container not found')
}

const root = ReactDOM.createRoot(container)

root.render(
  <React.StrictMode>
    <App />
  </React.StrictMode>,
)
```

`createRoot` tells React which part of the page it controls. After that, React renders your component tree inside that root element.

### Example 4 — What the compiled output of JSX looks like

JSX is not HTML. It is syntax that gets compiled into JavaScript.

```tsx
const element = <h1 className="title">Hello</h1>
```

```js
const element = React.createElement(
  'h1',
  { className: 'title' },
  'Hello',
)
```

You do not usually write `React.createElement` yourself, but JSX makes more sense once you know that JSX is just a nicer way to describe UI in JavaScript.

### Example 5 — A simple component tree

Real React apps are built as trees of components.

```tsx
function Header() {
  return <header>Header</header>
}

function Main() {
  return <main>Main content</main>
}

function Footer() {
  return <footer>Footer</footer>
}

export default function App() {
  return (
    <>
      <Header />
      <Main />
      <Footer />
    </>
  )
}
```

This is component-based architecture in its simplest form. `App` is made of smaller reusable UI pieces instead of one giant block of markup.

### Example 6 — Vite project setup

A beginner React app is often created with Vite because it sets up the tooling for you quickly.

```bash
npm create vite@latest react-intro-demo -- --template react
cd react-intro-demo
npm install
npm run dev
```

These commands create a React project, install dependencies, and start the development server. Vite handles module loading, JSX transformation, and fast local rebuilds for you.

### Example 7 — Folder structure explanation

A tiny React app usually starts with just a few important files.

```txt
react-intro-demo/
├── index.html
├── package.json
└── src/
    ├── main.jsx
    └── App.jsx
```

```html
<!-- index.html -->
<!doctype html>
<html lang="en">
  <head>
    <meta charset="UTF-8" />
    <meta name="viewport" content="width=device-width, initial-scale=1.0" />
    <title>React Intro</title>
  </head>
  <body>
    <div id="root"></div>
    <script type="module" src="/src/main.jsx"></script>
  </body>
</html>
```

```tsx
// src/main.jsx
import React from 'react'
import ReactDOM from 'react-dom/client'
import App from './App'

ReactDOM.createRoot(document.getElementById('root')).render(<App />)
```

```tsx
// src/App.jsx
export default function App() {
  return <h1>Hello React</h1>
}
```

`index.html` contains the root DOM element. `main.jsx` is the entry point that boots React. `App.jsx` contains your first real component and usually becomes the top of your component tree.

### Example 8 — The development server

When you run the development server, the tooling watches your files and rebuilds the app as you work.

```bash
npm run dev
```

That command starts a local server, transforms JSX into JavaScript, and refreshes the browser when files change. You focus on writing components, and the tooling handles the build step in the background.

## Common mistakes

<Callout type="mistake">
**Mistake:** Thinking React is a full framework that handles everything.
**Fix:** Treat React as the UI layer only.
**Why:** React focuses on rendering and component structure. Routing, data fetching strategy, authentication, and deployment conventions usually come from other tools.
</Callout>

<Callout type="mistake">
**Mistake:** Assuming React directly changes the DOM every time you write JSX.
**Fix:** Understand that JSX describes UI, and React decides how to update the real DOM.
**Why:** React first works with a JavaScript representation of the UI, then applies the needed real DOM changes.
</Callout>

<Callout type="mistake">
**Mistake:** Believing declarative UI means "less control" in a bad way.
**Fix:** Think of declarative UI as describing the final screen for the current data.
**Why:** You still control the logic. You just stop manually writing every DOM update step yourself.
</Callout>

<Callout type="mistake">
**Mistake:** Writing one giant component for the whole page.
**Fix:** Split the UI into smaller reusable components.
**Why:** Smaller components reduce duplication, make bugs easier to isolate, and make the codebase easier to maintain.
</Callout>

<Callout type="tip">
React is just JavaScript. Components are functions, props are arguments, and JSX becomes ordinary JavaScript after compilation. There is no hidden magic layer you need to fear.
</Callout>

<Callout type="note">
React does not handle routing, data fetching, or global state management by itself. Those concerns usually come from separate libraries or frameworks built around React.
</Callout>

<Callout type="tip">
Virtual DOM does not automatically mean "faster than everything else." The real benefit is more predictable updates and a better programming model for complex UI.
</Callout>

## How this is used in real projects

In a real product codebase, React is usually the layer that turns business data into actual screens. In an insurance SaaS application, you might receive policy data, claim data, billing data, and user permissions from APIs. React components turn that data into customer dashboards, claim timelines, billing summaries, approval forms, alerts, and navigation flows. Instead of manually editing DOM nodes whenever the data changes, the UI is re-described from the latest state and props.

Component-based architecture becomes even more valuable in production. A real page should not be one file with 800 lines of mixed markup and logic. A policy page might be split into components like `PolicyHeader`, `CoverageSummary`, `BillingSection`, `ActivityFeed`, and `RenewalNotice`. Each piece has one job. That makes code reviews easier, testing easier, and team collaboration much safer.

React is also valuable because it makes repeated UI patterns reusable. If your product uses the same button variants, table rows, badges, cards, empty states, and form controls across many screens, React components let you define those once and use them everywhere. That reduces duplication and creates a more consistent product.

## Interview questions

<Callout type="interview">
**Q:** What is the difference between React and a framework like Angular?
**A:** React is primarily a library focused on building user interfaces, especially the view layer. A framework like Angular gives you a broader, more opinionated structure with more built-in solutions for routing, forms, dependency injection, and application architecture. With React, you usually choose more of the surrounding stack yourself. With Angular, more decisions are made for you by the framework.
</Callout>

<Callout type="interview">
**Q:** What problem does the Virtual DOM solve?
**A:** The Virtual DOM gives React a structured way to compare what the UI used to look like with what it should look like now. Instead of forcing developers to manually track and perform every DOM update, React can calculate the necessary changes and apply them in a more predictable way. The biggest benefit is developer productivity and consistency in UI updates. It is not simply about raw speed in every scenario.
</Callout>

<Callout type="interview">
**Q:** What is React and why was it created?
**A:** React is a JavaScript library for building user interfaces. It was created to make dynamic UI easier to reason about, especially when many parts of the page depend on changing data. Before React, developers often updated the DOM manually, which became hard to maintain as applications grew. React improved that by letting developers describe the UI for the current state and letting React handle the updates.
</Callout>

<Callout type="interview">
**Q:** What is the Virtual DOM and how does it work?
**A:** The Virtual DOM is a lightweight JavaScript representation of the UI. When data changes, React creates a new virtual representation, compares it with the previous one, and figures out what changed. It then applies the necessary updates to the real browser DOM. This helps React manage UI changes in a predictable and structured way.
</Callout>

<Callout type="interview">
**Q:** What is the difference between declarative and imperative programming in UI?
**A:** Imperative UI means you tell the system step by step how to update the page, such as finding elements, changing text, or adding classes manually. Declarative UI means you describe what the UI should look like for the current data, and the library figures out how to make the page match that description. React uses the declarative model because it scales better as applications become more complex. It reduces the amount of manual DOM bookkeeping developers need to do.
</Callout>

<Callout type="interview">
**Q:** What is a Single Page Application and how is it different from a traditional website?
**A:** A Single Page Application loads one main HTML shell and then updates the visible content in the browser using JavaScript, usually without full page reloads for most interactions. A traditional website often requests a completely new HTML page from the server on each navigation. SPAs can feel more app-like and smooth once loaded. React is commonly used to build SPAs, although React itself does not define routing.
</Callout>

<Callout type="interview">
**Q:** Is React a framework or a library? What is the difference?
**A:** React is a library because it focuses mainly on one concern: building user interfaces. A framework usually provides broader structure, stronger conventions, and more built-in solutions across the application stack. In React, you often choose your own routing, state management, and data-fetching tools. In a framework like Next.js or Angular, more of those decisions are built in or strongly guided.
</Callout>
