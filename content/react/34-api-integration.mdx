## Why this exists

Fetching data is easy. Building a data integration layer that stays clean across a real application is much harder. At first, teams often place raw `fetch()` calls directly inside components. That works for a moment, but then problems pile up: duplicated headers, inconsistent error handling, fragile response parsing, secrets leaking into the wrong place, and no clear answer for retry, pagination, or validation.

This topic exists because advanced React work is not just about “getting data on the screen.” It is about building an API architecture that is secure, typed, reusable, and predictable as the app grows.

Good API integration architecture reduces duplication, protects secrets, makes failures easier to reason about, and gives the UI a stable contract instead of forcing every component to understand every raw network detail.

## How it works

The first principle is server-first API calling when secrets are involved. If an external API requires private credentials, those requests should happen on the server so API keys stay out of the browser. Client-side API calls are still valid in some cases, especially for user-triggered interactions or public endpoints, but they should be used intentionally rather than by default.

An API abstraction layer is the next major idea. Instead of calling `fetch()` directly everywhere, you wrap it in typed helper functions. Those helpers can centralize base URLs, auth headers, request options, response parsing, and error normalization.

Request and response normalization are important because different APIs often return different shapes. One endpoint may return `{ data: [...] }`, another may return `{ items: [...] }`, and another may return nested results. A good abstraction layer converts those differences into one consistent shape for the rest of the app.

Error normalization matters just as much. Some APIs return `message`, others return `error`, others return `errors[0].detail`. The UI should not need to know every external variation. Instead, one normalized `ApiError` shape can make error handling more consistent.

Secrets should be handled through environment variables and server-only modules. API keys should never live in the client bundle. If the API integration belongs on the server, keep it there clearly.

Pagination is another architectural choice. Offset pagination uses values like `page=2&limit=20`. Cursor pagination uses tokens like `after=abc123`. Offset pagination is simple and readable. Cursor pagination often scales better for large, frequently changing datasets.

Retry logic helps with temporary failures, but it should be deliberate. Some requests are safe to retry, others are not. A simple retry wrapper is useful for idempotent reads, but write operations need more care.

Rate limiting awareness matters when integrating third-party services. A UI or data layer that blindly retries or fires many duplicate requests can hit external limits quickly.

Typed API responses make all of this safer. TypeScript helps express the expected shape, and Zod can validate the actual runtime data before the UI trusts it. That combination gives you both editor-level guidance and runtime protection.

## Code examples

### Example 1 — API call in a Server Component

When secrets are involved, keep the request on the server.

```tsx
export default async function ReportsPage() {
  const apiKey = process.env.REPORTS_API_KEY

  if (!apiKey) {
    throw new Error('Missing reports API key')
  }

  const response = await fetch('https://api.example.com/reports', {
    headers: {
      Authorization: `Bearer ${apiKey}`,
    },
  })

  const reports = (await response.json()) as { id: number; title: string }[]

  return (
    <ul>
      {reports.map((report) => (
        <li key={report.id}>{report.title}</li>
      ))}
    </ul>
  )
}
```

### Example 2 — API abstraction in `lib/api.ts`

Centralize request logic instead of duplicating it across components.

```ts
const API_BASE_URL = 'https://api.example.com'

export async function apiRequest<T>(
  path: string,
  init?: RequestInit,
): Promise<T> {
  const response = await fetch(`${API_BASE_URL}${path}`, init)

  if (!response.ok) {
    throw new Error(`API request failed with status ${response.status}`)
  }

  return (await response.json()) as T
}

export function getInvoices() {
  return apiRequest<{ id: number; total: number }[]>('/invoices')
}
```

### Example 3 — Error normalization with a consistent `ApiError`

One stable error shape keeps UI code simpler.

```ts
export type ApiError = {
  message: string
  status: number
  code?: string
}

export async function parseApiError(response: Response): Promise<ApiError> {
  const body = (await response.json().catch(() => null)) as
    | { message?: string; code?: string }
    | null

  return {
    message: body?.message ?? 'Something went wrong',
    status: response.status,
    code: body?.code,
  }
}
```

### Example 4 — Typed response with Zod validation

TypeScript describes the intent. Zod checks the runtime value.

```ts
import { z } from 'zod'

const lessonSchema = z.object({
  id: z.number(),
  title: z.string(),
  completed: z.boolean(),
})

const lessonsSchema = z.array(lessonSchema)

export type Lesson = z.infer<typeof lessonSchema>

export async function getLessons(): Promise<Lesson[]> {
  const response = await fetch('https://api.example.com/lessons')
  const json = await response.json()

  return lessonsSchema.parse(json)
}
```

### Example 5 — Pagination patterns

Both offset and cursor pagination matter in real systems.

```ts
// Offset pagination
export async function getInvoicesPage(page: number, limit: number) {
  return apiRequest<{ items: Invoice[]; total: number }>(
    `/invoices?page=${page}&limit=${limit}`,
  )
}
```

```ts
// Cursor pagination
export async function getInvoicesAfter(cursor?: string) {
  const query = cursor ? `?after=${cursor}` : ''
  return apiRequest<{ items: Invoice[]; nextCursor: string | null }>(
    `/invoices${query}`,
  )
}
```

### Example 6 — Retry wrapper

Retries should be small, explicit, and limited.

```ts
export async function retry<T>(
  task: () => Promise<T>,
  attempts = 3,
): Promise<T> {
  let lastError: unknown

  for (let attempt = 1; attempt <= attempts; attempt += 1) {
    try {
      return await task()
    } catch (error) {
      lastError = error
    }
  }

  throw lastError
}
```

```ts
const lessons = await retry(() => getLessons(), 2)
```

### Example 7 — Complete typed API client

This is the kind of boundary a real app can build on.

```ts
import 'server-only'
import { z } from 'zod'

const projectSchema = z.object({
  id: z.string(),
  name: z.string(),
  status: z.enum(['active', 'archived']),
})

const projectsResponseSchema = z.object({
  data: z.array(projectSchema),
  nextCursor: z.string().nullable(),
})

type Project = z.infer<typeof projectSchema>

type ProjectsResponse = z.infer<typeof projectsResponseSchema>

type ApiError = {
  message: string
  status: number
}

async function request<T>(
  path: string,
  schema: z.ZodSchema<T>,
): Promise<T> {
  const token = process.env.EXTERNAL_PROJECTS_API_TOKEN

  if (!token) {
    throw new Error('Missing external API token')
  }

  const response = await fetch(`https://api.example.com${path}`, {
    headers: {
      Authorization: `Bearer ${token}`,
    },
  })

  if (!response.ok) {
    const error: ApiError = {
      message: 'External API request failed',
      status: response.status,
    }

    throw error
  }

  const json = await response.json()
  return schema.parse(json)
}

export async function getProjects(cursor?: string): Promise<ProjectsResponse> {
  const query = cursor ? `?after=${cursor}` : ''
  return request(`/projects${query}`, projectsResponseSchema)
}

export type { Project }
```

### Example 8 — Debounced search with abort logic

Real integrations often need to avoid stale requests and wasted work.

```tsx
import { useEffect, useState } from 'react'

function SearchBox() {
  const [query, setQuery] = useState('')
  const [results, setResults] = useState<string[]>([])

  useEffect(() => {
    if (!query.trim()) {
      setResults([])
      return
    }

    const controller = new AbortController()
    const timeoutId = window.setTimeout(async () => {
      const response = await fetch(`/api/search?q=${encodeURIComponent(query)}`, {
        signal: controller.signal,
      })

      const data = (await response.json()) as string[]
      setResults(data)
    }, 300)

    return () => {
      controller.abort()
      window.clearTimeout(timeoutId)
    }
  }, [query])

  return (
    <div>
      <input value={query} onChange={(event) => setQuery(event.target.value)} />
      <ul>{results.map((result) => <li key={result}>{result}</li>)}</ul>
    </div>
  )
}
```

## Common mistakes

<Callout type="mistake">
**Mistake:** Calling secret-dependent external APIs directly from the client.
**Fix:** Move those calls to server code.
**Why:** Secrets and private auth headers must stay off the browser.
</Callout>

<Callout type="mistake">
**Mistake:** Letting every component parse raw API error shapes differently.
**Fix:** Normalize errors in one API layer.
**Why:** Consistent error handling makes UI code simpler and less fragile.
</Callout>

<Callout type="tip">
Typed wrappers plus Zod validation are a strong combination. TypeScript helps author the code, and Zod helps protect runtime boundaries.
</Callout>

<Callout type="note">
Not every request needs retry logic. Retrying safe reads is common. Retrying writes can be dangerous unless the operation is idempotent.
</Callout>

## How this is used in real projects

In production apps, API integration is usually treated as infrastructure, not just a helper function. Teams often build one layer that owns base URLs, tokens, retry behavior, error normalization, and validation. Then route components or features call that layer instead of building requests from scratch every time.

This becomes especially important when multiple external services are involved. Billing APIs, analytics providers, search providers, and internal backend services all behave a little differently. A good integration layer smooths those differences so the rest of the React codebase can stay focused on UI and product logic.

## Interview questions

<Callout type="interview">
**Q:** Why do you call external APIs from Server Components instead of Client Components?
**A:** Because server-side calls can safely use secrets, private auth headers, database access, and internal network boundaries that should never be exposed to the browser. They also reduce the amount of client-side fetching logic needed in the UI.
</Callout>

<Callout type="interview">
**Q:** What is API abstraction and why is it useful?
**A:** API abstraction means wrapping low-level request logic in shared typed functions instead of calling `fetch()` directly everywhere. It is useful because it centralizes auth headers, parsing, error handling, validation, and request conventions so the UI layer stays cleaner and more consistent.
</Callout>

<Callout type="interview">
**Q:** How do you keep API keys secure in Next.js or other React-based full-stack apps?
**A:** Keep them in server-only environment variables, use them only in server code, and avoid exposing them through client bundles or `NEXT_PUBLIC_` variables. Requests that need those keys should go through server components, route handlers, or server-side integration layers.
</Callout>
