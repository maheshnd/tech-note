## Why this exists

React rendering is supposed to be pure. A component takes props and state, returns JSX, and describes what the UI should look like. But real applications also need to do things outside that pure rendering process. They need to fetch data, start timers, listen for browser events, talk to localStorage, and subscribe to outside systems. Those are side effects.

`useEffect` exists so React components can run side effects safely after rendering. Without it, developers would often try to put side-effect logic directly inside render, which leads to repeated requests, broken event listeners, memory leaks, and confusing bugs. `useEffect` gives React a structured place for this work and a structured way to clean it up.

## How it works

A side effect is anything that reaches outside React's normal render calculation. Examples include making a network request, starting a timer, writing to localStorage, or attaching a `window` event listener. Rendering JSX is not a side effect. Calculating derived values from state is not a side effect. Side effects are about synchronizing the component with something outside React.

`useEffect` runs after React has rendered the component and committed the changes to the DOM. This matters because side effects often need the rendered UI to exist first, or they need to happen after the screen updates. The effect callback does not run during render. It runs after render.

The dependency array controls when the effect runs:

- No dependency array: the effect runs after every render.
- Empty array `[]`: the effect runs once after the initial mount.
- Specific dependencies like `[query, page]`: the effect runs after mount and again whenever one of those values changes.

Effects can also return a cleanup function. Cleanup runs before the effect runs again and when the component unmounts. This is essential for things like intervals, event listeners, subscriptions, and in-flight async work. If you forget cleanup, you often create memory leaks or duplicate behavior.

You can think about mount, update, and unmount like this:

- Mount: the component appears for the first time, so mount-time effects run.
- Update: props or state change, the component renders again, and effects whose dependencies changed run again.
- Unmount: the component is removed, so cleanup runs one last time.

`useEffect` is powerful, but it is also easy to misuse. A missing dependency can create stale closure bugs where the effect keeps reading old values. A missing cleanup can leak intervals or listeners. A `setState` call inside an effect without the right dependencies can create an infinite loop. And some logic should not use `useEffect` at all, especially derived state or logic that should happen directly inside an event handler.

## Syntax

```tsx
useEffect(() => {
  // side effect logic

  return () => {
    // cleanup logic
  }
}, [dependency])
```

## Code examples

### Example 1 — Basic `useEffect` with no dependency array

If you do not provide a dependency array, the effect runs after every render.

```tsx
import { useEffect, useState } from 'react'

export default function RenderLogger() {
  const [count, setCount] = useState(0)

  useEffect(() => {
    console.log('Effect ran after render')
  })

  return <button onClick={() => setCount(count + 1)}>{count}</button>
}
```

This runs after the first render and after every later render too.

### Example 2 — `useEffect` with an empty dependency array

An empty array means "run once after mount."

```tsx
import { useEffect } from 'react'

export default function MountOnlyEffect() {
  useEffect(() => {
    console.log('Runs once after mount')
  }, [])

  return <p>Mounted component</p>
}
```

This is often used for one-time setup logic, though you still need cleanup if setup creates something that must be removed.

### Example 3 — `useEffect` with a dependency

When the effect depends on a value, put that value in the dependency array.

```tsx
import { useEffect, useState } from 'react'

export default function QueryLogger() {
  const [query, setQuery] = useState('')

  useEffect(() => {
    console.log('Query changed:', query)
  }, [query])

  return (
    <input
      value={query}
      onChange={(event) => setQuery(event.target.value)}
      placeholder="Search"
    />
  )
}
```

The effect runs after mount and then again only when `query` changes.

### Example 4 — Cleanup with `clearInterval`

Intervals must be cleaned up or they keep running after the component should be done with them.

```tsx
import { useEffect, useState } from 'react'

export default function Timer() {
  const [seconds, setSeconds] = useState(0)

  useEffect(() => {
    const intervalId = window.setInterval(() => {
      setSeconds((value) => value + 1)
    }, 1000)

    return () => {
      window.clearInterval(intervalId)
    }
  }, [])

  return <p>Seconds: {seconds}</p>
}
```

The cleanup function prevents the interval from leaking after unmount.

### Example 5 — Cleanup with `removeEventListener`

Browser listeners also need cleanup.

```tsx
import { useEffect, useState } from 'react'

export default function WindowWidth() {
  const [width, setWidth] = useState(window.innerWidth)

  useEffect(() => {
    function handleResize() {
      setWidth(window.innerWidth)
    }

    window.addEventListener('resize', handleResize)

    return () => {
      window.removeEventListener('resize', handleResize)
    }
  }, [])

  return <p>Window width: {width}</p>
}
```

Without cleanup, the listener would stay attached even after the component is gone.

### Example 6 — Fetching data on mount

Fetching data is one of the most common use cases for `useEffect`.

```tsx
import { useEffect, useState } from 'react'

type User = {
  id: number
  name: string
}

export default function UserList() {
  const [users, setUsers] = useState<User[]>([])
  const [loading, setLoading] = useState(true)
  const [error, setError] = useState('')

  useEffect(() => {
    async function loadUsers() {
      try {
        setLoading(true)
        setError('')

        const response = await fetch('https://jsonplaceholder.typicode.com/users')
        const data = (await response.json()) as User[]

        setUsers(data)
      } catch {
        setError('Failed to load users')
      } finally {
        setLoading(false)
      }
    }

    void loadUsers()
  }, [])

  if (loading) return <p>Loading users...</p>
  if (error) return <p>{error}</p>

  return (
    <ul>
      {users.map((user) => (
        <li key={user.id}>{user.name}</li>
      ))}
    </ul>
  )
}
```

This is the classic loading, error, data pattern.

### Example 7 — Fetching data when a dependency changes

Sometimes the effect should rerun whenever an input value changes.

```tsx
import { useEffect, useState } from 'react'

type SearchResult = {
  id: number
  title: string
}

export default function SearchResults() {
  const [query, setQuery] = useState('react')
  const [results, setResults] = useState<SearchResult[]>([])

  useEffect(() => {
    async function search() {
      const response = await fetch(
        `https://jsonplaceholder.typicode.com/posts?title_like=${query}`,
      )
      const data = (await response.json()) as SearchResult[]
      setResults(data)
    }

    void search()
  }, [query])

  return (
    <section>
      <input
        value={query}
        onChange={(event) => setQuery(event.target.value)}
      />
      <ul>
        {results.map((result) => (
          <li key={result.id}>{result.title}</li>
        ))}
      </ul>
    </section>
  )
}
```

The effect reruns whenever `query` changes because `query` is in the dependency array.

### Example 8 — Race condition problem

A race condition happens when an older request finishes after a newer request and overwrites newer data.

```tsx
import { useEffect, useState } from 'react'

export default function RaceConditionExample() {
  const [query, setQuery] = useState('')
  const [result, setResult] = useState('')

  useEffect(() => {
    async function search() {
      const response = await fetch(`/api/search?q=${query}`)
      const data = (await response.json()) as { value: string }
      setResult(data.value)
    }

    void search()
  }, [query])

  return (
    <section>
      <input value={query} onChange={(event) => setQuery(event.target.value)} />
      <p>{result}</p>
    </section>
  )
}
```

If the request for `rea` finishes after the request for `react`, the old response can incorrectly replace the new one.

### Example 9 — `AbortController` to cancel previous fetches

Abort controllers help prevent outdated requests from winning.

```tsx
import { useEffect, useState } from 'react'

type SearchResult = {
  id: number
  title: string
}

export default function SafeSearchResults() {
  const [query, setQuery] = useState('react')
  const [results, setResults] = useState<SearchResult[]>([])

  useEffect(() => {
    const controller = new AbortController()

    async function search() {
      try {
        const response = await fetch(
          `https://jsonplaceholder.typicode.com/posts?title_like=${query}`,
          { signal: controller.signal },
        )
        const data = (await response.json()) as SearchResult[]
        setResults(data)
      } catch (error) {
        if ((error as Error).name !== 'AbortError') {
          console.error(error)
        }
      }
    }

    void search()

    return () => {
      controller.abort()
    }
  }, [query])

  return (
    <section>
      <input value={query} onChange={(event) => setQuery(event.target.value)} />
      <ul>
        {results.map((result) => (
          <li key={result.id}>{result.title}</li>
        ))}
      </ul>
    </section>
  )
}
```

Cleanup aborts the previous request when the dependency changes again.

### Example 10 — Stale closure problem

Effects can accidentally capture old values if dependencies are wrong.

```tsx
import { useEffect, useState } from 'react'

export default function StaleClosureExample() {
  const [count, setCount] = useState(0)

  useEffect(() => {
    const intervalId = window.setInterval(() => {
      console.log('Count is:', count)
    }, 1000)

    return () => {
      window.clearInterval(intervalId)
    }
  }, [])

  return <button onClick={() => setCount((value) => value + 1)}>{count}</button>
}
```

Because the effect runs only once, it captures the first `count` value and keeps logging that stale value.

### Example 11 — Stale closure fix

Including the dependency fixes the stale captured value.

```tsx
import { useEffect, useState } from 'react'

export default function FixedClosureExample() {
  const [count, setCount] = useState(0)

  useEffect(() => {
    const intervalId = window.setInterval(() => {
      console.log('Count is:', count)
    }, 1000)

    return () => {
      window.clearInterval(intervalId)
    }
  }, [count])

  return <button onClick={() => setCount((value) => value + 1)}>{count}</button>
}
```

Now the effect reruns when `count` changes, so it sees the latest value.

### Example 12 — Infinite loop problem

An effect that sets state after every render can trigger an endless cycle.

```tsx
import { useEffect, useState } from 'react'

export default function InfiniteLoopExample() {
  const [count, setCount] = useState(0)

  useEffect(() => {
    setCount((value) => value + 1)
  })

  return <p>{count}</p>
}
```

This rerenders, reruns the effect, updates state again, and keeps going forever.

### Example 13 — Infinite loop fix

The solution depends on the goal, but the important part is using the correct dependency behavior.

```tsx
import { useEffect, useState } from 'react'

export default function FixedLoopExample() {
  const [count, setCount] = useState(0)

  useEffect(() => {
    setCount(1)
  }, [])

  return <p>{count}</p>
}
```

The empty dependency array means the effect runs only once after mount, so it does not loop forever.

### Example 14 — localStorage sync

Effects are often used to synchronize state with browser storage.

```tsx
import { useEffect, useState } from 'react'

export default function SavedTheme() {
  const [theme, setTheme] = useState('light')

  useEffect(() => {
    const savedTheme = window.localStorage.getItem('theme')

    if (savedTheme) {
      setTheme(savedTheme)
    }
  }, [])

  useEffect(() => {
    window.localStorage.setItem('theme', theme)
  }, [theme])

  return (
    <button onClick={() => setTheme(theme === 'light' ? 'dark' : 'light')}>
      Theme: {theme}
    </button>
  )
}
```

One effect reads on mount, and another writes whenever the state changes.

### Example 15 — Window resize event listener

Effects are a clean place to attach and remove browser listeners.

```tsx
import { useEffect, useState } from 'react'

export default function ResizeWatcher() {
  const [width, setWidth] = useState(window.innerWidth)

  useEffect(() => {
    function handleResize() {
      setWidth(window.innerWidth)
    }

    window.addEventListener('resize', handleResize)

    return () => {
      window.removeEventListener('resize', handleResize)
    }
  }, [])

  return <p>Width: {width}</p>
}
```

This keeps the component synchronized with the browser window.

### Example 16 — `setInterval` with cleanup

Timers are another common synchronization case.

```tsx
import { useEffect, useState } from 'react'

export default function ElapsedTime() {
  const [seconds, setSeconds] = useState(0)

  useEffect(() => {
    const intervalId = window.setInterval(() => {
      setSeconds((value) => value + 1)
    }, 1000)

    return () => {
      window.clearInterval(intervalId)
    }
  }, [])

  return <p>Elapsed: {seconds}s</p>
}
```

Without cleanup, the interval would continue running and leaking work.

### Example 17 — Subscription pattern

Subscriptions should be connected in the effect and disconnected in cleanup.

```tsx
import { useEffect } from 'react'

type Subscription = {
  unsubscribe: () => void
}

function subscribeToNotifications(callback: () => void): Subscription {
  callback()

  return {
    unsubscribe() {
      console.log('Unsubscribed')
    },
  }
}

export default function NotificationsPanel() {
  useEffect(() => {
    const subscription = subscribeToNotifications(() => {
      console.log('Subscribed')
    })

    return () => {
      subscription.unsubscribe()
    }
  }, [])

  return <p>Notifications active</p>
}
```

This is the common setup-cleanup subscription pattern.

### Example 18 — Async code inside `useEffect`

The effect callback itself should not be `async`. Define an async function inside and call it.

```tsx
import { useEffect, useState } from 'react'

type Post = {
  id: number
  title: string
}

export default function PostList() {
  const [posts, setPosts] = useState<Post[]>([])

  useEffect(() => {
    async function loadPosts() {
      const response = await fetch('https://jsonplaceholder.typicode.com/posts')
      const data = (await response.json()) as Post[]
      setPosts(data.slice(0, 5))
    }

    void loadPosts()
  }, [])

  return (
    <ul>
      {posts.map((post) => (
        <li key={post.id}>{post.title}</li>
      ))}
    </ul>
  )
}
```

This is the correct async pattern because the effect callback stays synchronous and can still return cleanup if needed.

### Example 19 — When NOT to use `useEffect`

Derived state should usually be calculated during render, not synchronized through an effect.

```tsx
import { useEffect, useState } from 'react'

// Wrong: storing derived data in state
export function WrongDerivedState() {
  const [firstName, setFirstName] = useState('Mahesh')
  const [lastName, setLastName] = useState('K')
  const [fullName, setFullName] = useState('')

  useEffect(() => {
    setFullName(`${firstName} ${lastName}`)
  }, [firstName, lastName])

  return (
    <section>
      <button onClick={() => setFirstName('React')}>Rename</button>
      <p>{fullName}</p>
    </section>
  )
}

// Right: derive it directly during render
export function RightDerivedState() {
  const [firstName, setFirstName] = useState('Mahesh')
  const [lastName] = useState('K')

  const fullName = `${firstName} ${lastName}`

  return (
    <section>
      <button onClick={() => setFirstName('React')}>Rename</button>
      <p>{fullName}</p>
    </section>
  )
}
```

If a value can be computed from existing state, compute it instead of syncing it through an effect.

### Example 20 — A complete search example with debounce and abort

This combines dependency-driven fetching, debounce timing, cleanup, and abort cancellation.

```tsx
import { useEffect, useState } from 'react'

type SearchResult = {
  id: number
  title: string
}

export default function DebouncedSearch() {
  const [query, setQuery] = useState('')
  const [results, setResults] = useState<SearchResult[]>([])
  const [loading, setLoading] = useState(false)
  const [error, setError] = useState('')

  useEffect(() => {
    if (!query.trim()) {
      setResults([])
      setLoading(false)
      setError('')
      return
    }

    const controller = new AbortController()
    const timeoutId = window.setTimeout(() => {
      async function runSearch() {
        try {
          setLoading(true)
          setError('')

          const response = await fetch(
            `https://jsonplaceholder.typicode.com/posts?title_like=${query}`,
            { signal: controller.signal },
          )
          const data = (await response.json()) as SearchResult[]
          setResults(data.slice(0, 5))
        } catch (caughtError) {
          if ((caughtError as Error).name !== 'AbortError') {
            setError('Search failed')
          }
        } finally {
          setLoading(false)
        }
      }

      void runSearch()
    }, 300)

    return () => {
      window.clearTimeout(timeoutId)
      controller.abort()
    }
  }, [query])

  return (
    <section>
      <input
        value={query}
        onChange={(event) => setQuery(event.target.value)}
        placeholder="Search posts"
      />

      {loading ? <p>Searching...</p> : null}
      {error ? <p>{error}</p> : null}

      <ul>
        {results.map((result) => (
          <li key={result.id}>{result.title}</li>
        ))}
      </ul>
    </section>
  )
}
```

This is a strong real-world effect example because it handles cleanup, async timing, network cancellation, and multiple UI states together.

## Common mistakes

<Callout type="mistake">
**Mistake:** Making the effect callback `async` directly.
**Fix:** Define an async function inside the effect and call it.
**Why:** The effect callback itself should return either nothing or a cleanup function, not a Promise.
</Callout>

<Callout type="mistake">
**Mistake:** Forgetting cleanup for `setInterval`.
**Fix:** Return a cleanup function that calls `clearInterval`.
**Why:** Without cleanup, intervals keep running after the component should be done with them, causing memory leaks and duplicate behavior.
</Callout>

<Callout type="mistake">
**Mistake:** Forgetting cleanup for event listeners.
**Fix:** Return a cleanup function that removes the listener.
**Why:** Missing cleanup can leave old listeners attached, which causes leaks and repeated event handling.
</Callout>

<Callout type="mistake">
**Mistake:** Putting an object or array directly in the dependency array when that value is recreated every render.
**Fix:** Memoize the value or depend on the specific primitive fields you actually need.
**Why:** A new reference every render makes React think the dependency changed every time, which can cause endless reruns.
</Callout>

<Callout type="mistake">
**Mistake:** Leaving dependencies out of the array.
**Fix:** Include every reactive value the effect uses, unless you intentionally restructure the code to avoid that dependency.
**Why:** Missing dependencies often create stale closure bugs where the effect keeps using old values.
</Callout>

<Callout type="mistake">
**Mistake:** Putting too much unrelated logic into one `useEffect`.
**Fix:** Split effects by concern.
**Why:** Smaller focused effects are easier to understand, test, and debug than one large effect doing several unrelated jobs.
</Callout>

<Callout type="tip">
If the hooks lint rule warns about missing dependencies, fix the dependency problem instead of disabling the lint rule.
</Callout>

<Callout type="tip">
Think of effects as synchronization: "keep this external thing in sync with this state or prop." That mental model is often clearer than thinking only in lifecycle terms.
</Callout>

<Callout type="warning">
Calling `setState` inside an effect without the correct dependency behavior can create infinite loops.
</Callout>

<Callout type="interview">
**Q:** What is a stale closure in `useEffect`?
**A:** A stale closure happens when an effect captures an old value from a previous render and keeps using it because the dependency array prevented the effect from updating. For example, an interval started in an effect with `[]` may keep reading the first `count` value forever. This happens because the callback closes over the values that existed when the effect ran. The fix is usually to add the missing dependency or restructure the logic so it always sees the latest value.
</Callout>

<Callout type="interview">
**Q:** What is a race condition in data fetching?
**A:** A race condition happens when multiple async requests are in flight and an older request finishes after a newer one, incorrectly overwriting the newer result. This often appears in search UIs or dependency-driven fetches where the user types quickly. The common fixes are cleanup guards, request version tracking, or `AbortController` cancellation. The goal is to ensure only the latest relevant request updates the UI.
</Callout>

## How this is used in real projects

In real applications, `useEffect` is where React components synchronize with the outside world. A page may fetch data when filters change, subscribe to browser events, write preferences into localStorage, or open and close timers. That makes `useEffect` incredibly important, but also easy to overuse. Many React bugs in production come from effects that were missing cleanup, missing dependencies, or handling work that should not have been an effect at all.

Well-designed applications usually keep effects focused and intentional. One effect may handle data fetching. Another may synchronize a saved preference. Another may attach a browser listener. Breaking effects apart by purpose makes the code much easier to reason about than one huge effect that mixes networking, timers, logging, and DOM logic together.

The best React code also knows when not to use `useEffect`. Derived data, straightforward event logic, and purely render-time calculations usually belong outside effects. That restraint matters because unnecessary effects create extra state transitions, extra renders, and harder-to-debug code.

## Interview questions

<Callout type="interview">
**Q:** What is a side effect in React?
**A:** A side effect is any operation that reaches outside React's normal rendering process. Examples include fetching data, writing to localStorage, starting a timer, or attaching an event listener. Rendering JSX itself is not a side effect. `useEffect` exists to run these outside interactions safely after React renders.
</Callout>

<Callout type="interview">
**Q:** What are the three forms of the dependency array and what does each do?
**A:** No dependency array means the effect runs after every render. An empty array `[]` means the effect runs once after mount. A dependency array like `[query, page]` means the effect runs after mount and again whenever one of those listed values changes. This is how you control when the effect resynchronizes.
</Callout>

<Callout type="interview">
**Q:** What is the cleanup function and when does it run?
**A:** The cleanup function is the function returned from inside a `useEffect`. It runs before the effect runs again and when the component unmounts. Cleanup is used to stop timers, remove listeners, cancel subscriptions, and abort in-flight work. It prevents memory leaks and duplicated side effects.
</Callout>

<Callout type="interview">
**Q:** What is a stale closure in the context of `useEffect`?
**A:** A stale closure happens when an effect callback keeps using old values from the render where the effect was created. This often happens when dependencies are missing from the array. The effect then keeps reading outdated state or props instead of the latest ones. The fix is to include the right dependencies or refactor the logic to avoid stale captured values.
</Callout>

<Callout type="interview">
**Q:** What is a race condition in data fetching and how do you prevent it?
**A:** A race condition happens when multiple fetches overlap and an older request finishes after a newer one, overwriting fresh data with stale data. It often happens in search UIs or dependency-driven requests. You prevent it by aborting outdated requests, tracking which request is current, or guarding updates in cleanup logic. `AbortController` is one of the most common fixes.
</Callout>

<Callout type="interview">
**Q:** When should you NOT use `useEffect`?
**A:** You should not use `useEffect` for values that can be derived directly during render, for logic that belongs inside an event handler, or for work that does not need synchronization with something outside React. Many beginners overuse `useEffect` for derived state and end up creating unnecessary extra renders. If something can be computed directly from props and state, it usually should be. Effects are best for synchronization, not for every piece of logic.
</Callout>

<Callout type="interview">
**Q:** How do you correctly write async code inside `useEffect`?
**A:** The effect callback itself should not be marked `async`. Instead, define an async function inside the effect and call it. This keeps the effect callback synchronous so it can still return cleanup normally. That is the standard pattern for fetches and other async work inside `useEffect`.
</Callout>

<Callout type="interview">
**Q:** Why does putting an object in the dependency array cause an infinite loop?
**A:** If the object is recreated on every render, its identity changes on every render even if its contents look the same. React compares dependencies by reference, so it sees the object as new each time and reruns the effect. If that effect then updates state, the component rerenders and recreates the object again, which can loop forever. The fix is to memoize the object or depend on stable primitive values instead.
</Callout>
